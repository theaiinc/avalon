- The repository lives on the Data volume at `/Volumes/Data/llama-dash`.
- `/Users/stevetran/llama-dash` is a symlink to `/Volumes/Data/llama-dash` for compatibility with existing workspace references.
- The project is named Avalon. The frontend logo asset is `frontend/src/assets/avalon-logo.png`, copied from `/Users/stevetran/Downloads/avalon-logo.png`.
- Backend storage is configured in `backend/config.py`; it resolves to the repo-local `data/` directory (`/Volumes/Data/llama-dash/data`).
- GPU detection in `backend/gpu_detector.py` covers NVIDIA (`nvidia-smi`), macOS Metal via `system_profiler SPDisplaysDataType`, Windows Intel/NPUs via PowerShell, and Vulkan (`vulkaninfo`). On this machine the Metal detector reports `Apple M3 Pro` with Metal 4 and uses `sysctl -n hw.memsize` for unified memory.
- Agent-facing interfaces live in `backend/agent_cli.py` and `backend/mcp_server.py`. The CLI emits JSON and wraps the dashboard REST API on port `8771`; run it with `cd backend && .venv/bin/python agent_cli.py gpu-list`. The MCP server is stdio-based; run it with `LLAMA_DASH_URL=http://127.0.0.1:8771 .venv/bin/python mcp_server.py`.
- The frontend has an Agents page at `/agents` that shows MCP, CLI, and API connection commands for other agents.
- PC Links must use pairing-code flow, not manual URL-only linking: the local Avalon enters a pairing code, the remote Avalon completes pairing and returns/provides a unique API key for that linked PC. The API key must be saved in this local Avalon instance and used for all remote model/list/chat requests. Linked PC models appear in the local gateway as `pc:<pc_id>:<remote_model_id>`.
- The dashboard can start a separate local inference API on port `8787` with OpenAI-compatible `/v1/models` and `/v1/chat/completions` plus Anthropic-compatible `/v1/messages`; both the CLI and MCP server expose helpers for that lifecycle.
- The local inference API is a multi-model gateway. Downloaded models under `data/models/` automatically appear in `/v1/models`; callers select a model with the request `model` field. GGUF requests run via the active `llama-cli` backend driver; OpenVINO requests use cached `openvino_genai` pipelines.

GGUF models use `gguf_backend`/`gpu_index`, while OpenVINO models use `openvino_device`; keep legacy `device` only as a compatibility alias.
- Aha: keep the API Server Configuration card gateway-level only (port, API mode, start/stop). Do not put Quick Test model selection, GGUF backend buttons, or GPU device controls there; per-model serving details belong in served profiles and Quick Test owns its own model picker.

Profiles include `served_id`, MTP and dSpark booleans inferred from model/repo/file names, driver/runtime, model file, and serving configuration such as API mode, port, backend, OpenVINO device, GPU index, default max tokens, and default temperature.

Overrides are stored in `data/served_profiles.json` through `backend/served_profiles.py` and keyed by the original `base_id`, so edited `served_id` aliases can still route to the underlying local model. The API page's Model Catalog card owns profile edit/save/reset UI.
- Aha: the dashboard backend may restart while the port `8787` gateway keeps running. The gateway exposes `/v1/status`, and `backend/api_server.py` uses it to recover UI fields (`model_id`, `port`, `mode`, `device`, `gpu_index`) instead of showing blank defaults for an already-running API server.
- Aha: this llama.cpp `llama-cli` build auto-enters interactive conversation mode when a chat template exists. Gateway subprocess calls must pass `--single-turn`, otherwise HTTP completions hang until the 120s timeout after generating the answer. - Aha: OpenAI-compatible streaming must be implemented in Avalon itself for local GGUF models. `backend/api_server.py` now returns `text/event-stream` for `stream:true` by reading `llama-cli` stdout incrementally, filtering startup UI/log output, and suppressing `[Start thinking]...[End thinking]` so clients receive only final assistant tokens. The first visible token can still be delayed until the model finishes its thinking block.
- Aha: the API Server page can read live gateway state directly from `http://localhost:<port>/v1/status`; the child gateway enables CORS and attaches per-model `current_request`/`last_request` stream metrics. The streamed token count is an estimate from generated model stdout before the thinking filter, because `llama-cli` does not expose exact token IDs or token boundaries through stdout.
- Aha: per-model stream metrics include `first_token_at_ms` and `first_token_latency_ms` for TTFT. If a `current_request` stops updating for more than 15s, `/v1/status` marks it `stale`, moves it to `last_request`, and clears the current stream so the dashboard does not stay stuck on Streaming.
- Aha: do not mark a stream stale solely because no tokens arrived for 15s. Some `llama-cli` requests keep running while loading/thinking before first stdout output. Stream metrics now attach the subprocess PID and only stale-clear a quiet request when the process is no longer alive.

- Aha: GitHub release assets must be below 2 GiB; Linux release packaging therefore overrides electron-builder with gzip level 5, balancing the sidecar tarball size against the 45-minute build timeout.
- Aha: if the compressed Linux tarball still exceeds GitHub's 2 GiB per-asset limit, the release job splits it into 1900 MiB parts and publishes reconstruction instructions alongside the release manifest.
- Aha: the release watchdog must ignore its own job when checking completion; waiting for the workflow status to become completed creates a self-deadlock and eventual cancellation.
- Aha: packaged Electron pages load through `file://`; Vite must use `base: "./"` or production CSS and JavaScript resolve to `/assets/...` and the app renders as a blank window.
